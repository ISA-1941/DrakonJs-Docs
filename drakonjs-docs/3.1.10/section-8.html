Section 8. Binary Tree Algorithms

8.1. Representation of Binary Trees

A binary tree is a hierarchical data structure in which each node has at most two children, referred to as the left and right child. The topmost node is called the root, and nodes with no children are called leaves. Four types of imbalanced states exist.

Binary trees as an abstract data type were discussed in Section 1. Here we will discuss the computer implementation of this type of data structure using the visual algorithmic language DRAKON and the JavaScript programming language. Recall the basic terminology of binary trees (Figure 8.1):

Figure 8.1. Basic tree terminology

Root: The root of the tree is the only node with no incoming edges. It is the top node in the tree.
Node: The basic element of the tree. Each node has data and two references that can point to zero or more descendants.
Edge: A fundamental part of the tree used to connect two nodes.
Path: An ordered list of nodes connected by edges.
Leaf: A leaf node is a node that has no descendants.
Tree height: The number of edges on the longest path between the root and a leaf.
Node level: The number of edges on the path from the root node to that node.

Figure 8.2. Binary tree structure (value - value (key), N - NULL)

8.2. Binary Search Tree

8.2.1. Building a Binary Tree

If the tree is organized such that, for each node, all node values of its left subtree are less than this node and all values of its right subtree are greater, it is called a binary search tree (BST). A BST is a recursive data structure because each subtree is also a tree. A BST has the following properties:

Each node retains a unique value.
Each node has zero, one, or two child nodes.
One node is referred to as the root node, which is at the top of the tree structure.
Each node has only one parent node, except for the root node which has no parent.
The value of each node is greater than that of its left child but less than that of its right child.

A binary search tree is built according to a specific algorithm. Consider the sequence of integers {11, 5, 17, 15, 1, 8, 19, 13, 21}, represented as a slice of numbers. First the root node {11} is formed, then in a loop for each node the method insert(tree, value) is called recursively (Figure 8.3a), which in turn calls the method createNode(value), which creates a new node (Figure 8.3b).

Figure 8.3. Binary tree DRAKON-diagrams: a) building a tree by insert(node, value); b) creating a node

The complete process of creating a binary tree is shown in Figure 8.4. The first number 11 is written to the root of the tree. The second number 5 is less than the value in the tree root, so it is written to the left subtree. The next number 17 is greater than the root number, so it is written to the right subtree. Then the number 15 is greater than the value at the root of the tree, so it is written to the right subtree, but the right subtree is already built. The number 15 is compared to the number 17 at the root of the right subtree. Since the value to be added is less than the value in the root of the right subtree, we add the left subtree to this node. The final result is a binary search tree with three variants: node 5 is the parent of two children (1, 8), node 15 has only a left child, and node 19 has only a right child. This arrangement of nodes is chosen to demonstrate the node removal function, which will be discussed later.

8.2.4. Traversing the Binary Search Tree

One of the basic tree operations is traversing tree nodes. Unlike linear data structures, where elements are traversed in a linear order, tree nodes can be traversed in a variety of ways.

Pre-order walk: Each ancestor node is visited before its descendants.
In-order walk: The left subtree is visited from bottom to top, then the root node, then the right subtree.
Post-order walk: Descendants are visited first, then ancestors.

8.3. Self-Balancing Binary Trees (AVL Trees)

The efficiency of any tree operation depends substantially on the order in which the input data is received. For example, if an incoming sequence of numbers is partially sorted in ascending or descending order, the structure will no longer look like a tree (Figure 8.13).

Figure 8.13. Partially sorted input data

In such practically "degenerate" trees, the complexity of operations is determined by the number of nodes — that is, it is almost linear: O(n). The left and right subtrees are unbalanced, which can be estimated by the balance factor (bf), which is equal to the difference between the heights of the left and right subtrees. Recall that the height of a tree is defined as the length of the longest branch in a subtree (sum of edges). For an ideal binary search tree (a tree where the number of nodes in the left subtree equals the number of nodes in the right subtree), this factor is 0 (Figure 8.14).

Figure 8.14. An ideal binary tree

In real life, ideal binary trees are almost never achieved. Programmers often try to build a tree where the height of the left subtree differs from the height of the right subtree by no more than 1. Such trees are called AVL trees; for these trees the complexity of operations is defined as O(log n), meaning the execution time of basic operations (search, delete) is significantly less than for BST trees.

The algorithms for such trees are based on the process of balancing the tree when a new node is inserted or an existing node is removed. The purpose of balancing is to reconstruct the tree so that the heights of the left and right subtrees do not differ by more than 1. The balance factor must satisfy the following conditions:

The allowable values are bf = -1, 0, and +1.
bf = -1 indicates that the right subtree "outweighs" the left subtree.
bf = +1 indicates that the left subtree "outweighs" the right subtree.
bf = 0 indicates that the tree contains an equal number of nodes on each side — that is, the tree is perfectly balanced.

8.4. Red-Black Trees

8.4.1. Description of Red-Black Tree Properties as a Data Structure

A red-black tree is a variant of a self-balancing binary search tree in which the nodes are colored red or black according to a specific set of rules (Figure 8.29).

Figure 8.29. Red-black tree example

The nodes containing data (integers in this example) are called internal nodes. In addition, red-black trees contain imaginary "null" nodes associated with tree leaves, which are explicitly colored black (labeled "Null" in Figure 8.29). Red-black trees satisfy all properties of a binary search tree and must maintain the following specific invariants:

Each node is colored either red or black.
The root of the tree must be black. (Note: This invariant may be temporarily violated during the balancing process but must be restored before the operation completes.)
All leaves (NIL nodes) are black.
Both children of a red node must be black. Consequently, no two red nodes can appear consecutively on any path.
Every simple path from a node to any of its descendant leaves must contain the same number of black nodes.

Unlike AVL trees, where balance is enforced by limiting the height difference between subtrees, red-black tree balance is achieved by strictly enforcing the properties listed above. Inserting or removing a node can disrupt these invariants. Balance is restored using two primary operations: recoloring nodes and/or restructuring the tree (or subtrees) using specific rotations. The most critical of these constraints are Properties 4 and 5.

Property 4 mandates that two red nodes cannot be adjacent.
Property 5 builds on this by enforcing a uniform black height for all paths from any node to its leaf descendants. Regardless of the specific path taken, the count of black nodes remains constant.
Mechanical Analogy for Balancing: The Chandelier and the Balancing Machine

Understanding the dynamic enforcement of red-black properties is crucial for developing an efficient balancing algorithm. However, this often presents significant challenges, as it requires tracking the state of multiple nodes simultaneously.

To facilitate conceptual clarity, we employ a mechanical analogy. Since the "root" of this data structure is positioned at the top, a chandelier serves as an appropriate mechanical analogue — an elegant hanging structure where each node is a ball of a strictly defined color and weight, rigidly connected to form branches. The root is fixed at the top, and the weightless NIL leaves hang freely downward.

In this model:

Black balls are solid and heavy, providing stability and physical equilibrium. Their weight is defined as 1 unit.
Red balls are hollow, decorative, and virtually weightless (0 units). They serve as a flexible resource that can be "hardened" into black balls (weights) when necessary.

All NIL leaves are considered black and weightless termination points.

The structural equilibrium (balance) is defined by two simple conditions:

The total weight (the number of black balls) on any path from the root to any leaf must be equal.
Two red balls in a row are forbidden — this creates a "weak spot" in the chain, even though it does not alter the branch's overall weight.

Black height is the fundamental metric in this engineering model, representing the total "weight" (the number of black nodes) of a specific branch. Properties 4 and 5 work together to ensure structural integrity. While red nodes allow branches to stretch, the rules guarantee that the longest path (alternating black weights and red spacers) is never more than twice the length of the shortest path (consisting only of black weights). This constraint ensures that "tension" is evenly distributed, keeping the tree approximately balanced with a height of O(log n), where n is the number of nodes.

The entire lifecycle of the structure is managed by a virtual balancing automaton. It possesses an unlimited stock of spare black and red balls and performs four functions:

Insertion: The automaton hangs a new red ball (NIL replacement).
Deletion: It removes a ball and replaces it with a dummy, triggering rebalancing if necessary.
Balance Monitoring: After every insertion or deletion, it checks for equilibrium violations.
Balancing: If a violation is detected, the automaton performs local transformations: recoloring balls and/or rearranging connections (rotations) using its stock of spare parts.

To implement these operations, we define two custom data types: node and rbTree. A node includes fields for the key (numeric value), color (boolean: true for red, false for black), and pointers to left, right, and parent nodes. The rbTree structure is a reference-type variable that stores a pointer to the memory area containing the tree objects (Figure 8.30).

Cluster Node Notation

To analyze the balancing process, we focus on a local C-Cluster of nodes. This notation is essential for understanding the transformations applied by the automaton. The following designations are used (Figure 8.31):

Figure 8.31. Designation of "kinship" degrees of RB tree nodes (Cluster x, P, G, U, S)

In this model:

x: The newly inserted red node.
P: Parent of x.
G: Grandparent of x (parent of P). In our model, G is always a solid 'weight' (black) before fixup begins.
U: Uncle of x (sibling of P). A missing (null) uncle is treated as a static black 'weight' (NIL leaf).
S: Sibling of x (the other child of P). Its color and position are crucial for determining the necessary rotation type.

For a compact description of the cluster arrangement, we introduce the following symbolic notation:

A colon ":" separates the node name from its color (P:B describes a black parent).
The comparison symbols "<" and ">" indicate relationship direction (x<P means x is P's left child).
Commas and semicolons delimit attributes.

For example, the notation x:R, x<P; P:R, P>G; G:B; U:R; U>G; describes a specific "zig-zag" violation where red node x is the left child of its red parent P, which in turn is the right child of the black grandparent G, who also has a red uncle U. Cluster nodes can be in a "zig-zig" (e.g., x<P<G) or "zig-zag" (e.g., x<P>G) configuration.

This concise notation significantly simplifies the balancing algorithms, making the transformation criteria visual and efficient.

8.4.3. Deleting a Node: Restoring the Weight Balance

Node deletion in a red-black tree is a sophisticated operation because it must preserve the tree’s "gravitational" balance. However, by adhering to our analogy of weights (black nodes) and zero-gravity spacers (red nodes), the logic becomes intuitive: complexity only arises when we lose a "weight."

The standard mechanical notation for the C-Cluster used in deletion operations is shown in Figure 8.39:

z: The node to be physically deleted, which triggers the Weight Debt (the "Double-Black" focus).
P: Parent of *z*.
s: Sibling of *z*.
s.l, s.r: The left and right children of sibling *s* (the "nephews").

Figure 8.39. Cluster deletion nodes and their designations (z, P, s, s.l, s.r)

The complete deletion and rebalancing process is summarized in the mindmap in Figure 8.40.

Figure 8.40. Mindmap summarizing RB tree deletion operations

The operation consists of three primary steps:

Identification: As in any standard BST, the algorithm first locates the target node (*z*).
Structural Deletion and Replacement: We must physically remove the node in a way that minimizes damage:
If node *z* has at most one child: We remove *z* and replace it with its non-NIL child (or a NIL node).
If node *z* has two children: We cannot delete it directly. Instead, we find its inorder successor (the minimum node in its right subtree), swap their values, and then delete the successor. This ensures that the node being physically removed always has zero or one child.
The Color Impact (The Weight Factor): Rebalancing necessity is determined entirely by the color of the physically deleted component:
Deleting a Red Node: If the deleted node is red, no action is required. As red nodes are weightless, their removal does not alter any black-height paths, and Property 5 remains intact.
Deleting a Black Node: If the deleted node is black, we have a problem. We have removed a "brick" from one side of the scale. This branch is now "lighter," violating Property 5 and creating a Weight Deficit.

This deficit is the core of the "Double Black" problem. It signals to our balancing automaton that it must either:

  1. Find a spare red node in the vicinity and 'harden' it into a weight.

  2. Or redistribute weights from neighboring branches to fill the gap.

The "Double Black" Concept (The Weight Debt): The node that occupies the position of the physically deleted black node is temporarily designated as "Double Black" (DB). In our chandelier analogy, this is a Weight Debt — a conceptual void indicating the path through this node is "lighter" than Property 5 requires. Correcting this violation employs a sequence of recoloring and rotations:

Weight Transfer: Shifting a black weight from a neighboring branch via rotation.
Color Hardening: Transforming a red node into a black weight to fill the void.
Debt Propagation: If the local cluster cannot cancel the void, the "debt" is moved up toward the parent. The parent then becomes the new focus of the balancing process until the deficit is cancelled at the root or resolved by a rotation.

8.6. Conclusion

Trees are one of the most important data structures in computer science, used to represent hierarchical relationships and organize data. They provide efficient search, insert, and delete operations, making them ideal for use in a variety of applications.

AVL trees, named after their creators Adelson-Velsky and Landis, are balanced binary search trees. They provide fast search, insertion, and deletion by keeping the height of the tree logarithmic relative to the number of nodes. However, AVL trees require additional insertion and deletion operations to maintain balance, which can be a disadvantage in some applications.

Red-black trees are another type of balanced binary search tree. They provide efficient search, insert, and delete operations, but can be more difficult to implement because of the additional properties they must support. They are widely used in many standard libraries.

The choice of a specific tree type depends on the specific requirements of the application. AVL trees are generally preferred when frequent searches are the most important operations, as they provide a more stringent balance than red-black trees. On the other hand, red-black trees can be more effective with frequent insertions and removals, as they require fewer rotations to maintain balance.

It is important to remember that both types of trees require additional memory to store balance information, and both can be difficult to implement. In some cases, other data structures, such as hash tables or B-trees, may be more appropriate. However, despite these challenges, trees continue to play a central role in computer science and are a key tool for any programmer.
