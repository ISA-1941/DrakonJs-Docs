Section 9. Graphs and Graph Algorithms

9.1. General Information. Basic Terminology

This section discusses data structures such as graphs. In mathematics, a graph is a structure consisting of vertices (nodes) and edges (connections) that connect these vertices.

In general, graphs share many similarities with trees, and one can consider trees as a special case of graphs. However, the practical value of graphs in solving real-world problems is much greater. Many tasks can be reduced to considering a set of objects whose properties are described by the relationships between them.

Such objects include:

  • Electrical and electronic circuits

  • Printed circuit boards

  • Road maps

  • Aviation routes

  • Descriptions of structural constructions

  • Game states and logic

Among the tasks that can be solved using graphs are:

  • Finding the shortest path between two vertices

  • Solving the problem of maximum capacity of pipelines, road networks, or computer networks

  • Distributing N workers to perform M different types of work

  • Choosing the most efficient method of problem solving, among others.

Formally, a graph is denoted as $G = (V, E)$, where $V$ is the set of vertices and $E$ is the set of edges connecting them.

Directed and Undirected Graphs

Graphs as data structures can be represented in two primary forms: undirected and directed.

  1. Undirected Graph:: In these graphs, there are no "inputs" or "outputs"; there is only the objective fact of a connection.

    • Vertex Degree (deg(v)): This is the total number of edges incident to a given vertex.

    • Loops: In an undirected graph, a loop is counted twice (+2 to the degree). Why? Because both ends of the loop-edge are "plugged into" the same vertex, and each end contributes to the total degree.

  2. Directed Graph (Digraph):: Here, the logic becomes more nuanced as direction matters (Figure 9.1). The connection from vertex $u$ to vertex $v$ is not equivalent to a connection from $v$ to $u$. The degree is split into two distinct components:

    • In-degree (deg-(v)): The number of edges entering (incident into) the vertex.

    • Out-degree (deg+(v)): The number of edges leaving (incident from) the vertex.

    • Total Degree: The sum of the in-degree and out-degree.

    • Loops in a Digraph: In this case, a loop is counted exactly once for the in-degree and once for the out-degree. It does not "double" the total degree; instead, it is honestly distributed between the two directions.

Figure 9.1. Graphical representation of (a) an undirected graph; (b) a directed graph (digraph)

Adjacency, Incidence, and Degree

A vertex and an edge are incident if the vertex is one of the endpoints of the edge.

Refined Terms for Directed Graphs (Digraphs):

  • In-degree (deg-(v)): The number of edges directed into the vertex. In flow-based models, this represents the "input" capacity.

  • Out-degree (deg+(v)): The number of edges directed away from the vertex. In state machines, this represents the number of possible state transitions.

  • Total Degree: The sum of in-degree and out-degree.

  • Self-Loop (Digraph): A special case where an edge starts and ends at the same vertex. It contributes exactly 1 to the in-degree and exactly 1 to the out-degree.

9.2. Graph Representation Methods

Solving problems related to processing a data set organized in the form of graphs requires modeling them in computer programs. In practice, two structures are typically used to model the structure of a graph: the adjacency matrix and the adjacency list.

Adjacency Matrix

An adjacency matrix is a two-dimensional array that indicates the existence of a relationship between two vertices. If the graph has $V$ vertices, then the adjacency matrix is a $V \times V$ array. In an undirected graph, the adjacency relationship is symmetric: if vertex $u$ is adjacent to $v$, then $v$ is also adjacent to $u$. This symmetry is reflected in the adjacency matrix, which is always symmetric about its main diagonal.

In contrast, for a directed graph (or digraph), the relationship is not necessarily symmetric. An edge $(u, v)$ means vertex $v$ is adjacent from vertex $u$, and this is indicated in the matrix by a '1' at position A[u][v] only.

Figure 9.3. Adjacency matrix layout for undirected and directed graphs

Adjacency List

In some cases, a more memory-efficient way to represent a graph is to use an adjacency list. This method involves an array of linked lists (or nested arrays), where each individual list contains information about which vertices are adjacent to a given one.

Figure 9.4. Adjacency list structural representation

Comparison and Complexity

The space complexity of representing a graph depends heavily on the chosen data structure:

  • Adjacency Matrix: Requires $O(V^2)$ memory. This is memory-intensive, especially for large, sparse graphs (graphs with relatively few edges). It also often requires knowing the number of vertices in advance.

  • Adjacency List: Requires $O(V + E)$ memory, where $E$ is the number of edges. This is much more efficient for sparse graphs, which are common in real-world applications.

In practice, the choice between these representations is a trade-off. An adjacency matrix is often preferable for dense graphs (graphs where the number of edges approaches $V^2$) and when frequent edge lookups are required ($O(1)$ complexity). An adjacency list is generally better for sparse graphs.

Weighted Graphs

Many real-life problems require modeling graphs with numerical values associated with edges, such as distance, cost, or travel time. Such graphs are called weighted graphs. In an adjacency matrix, the weight is stored as the value of the matrix element (instead of '1'), and a special value (like $\infty$ or '0') can represent the absence of an edge. In an adjacency list, each element in the list is typically a structure containing both the adjacent vertex ID and the edge weight.

Figure 9.5. Adjacency list equipped with a 'weight' field

Graph Representation in JavaScript

Since JavaScript is a dynamic language, we rely on the flexibility of Objects, Arrays, and built-in collections.

Adjacency Matrix in JavaScript
// Initialize a 3x3 matrix for an undirected graph
const numVertices = 3;
const adjMatrix = Array(numVertices).fill().map(() => Array(numVertices).fill(0));

function addEdgeMatrix(u, v) {
    adjMatrix[u][v] = 1;
    adjMatrix[v][u] = 1; // For undirected graph
}

addEdgeMatrix(0, 1);
addEdgeMatrix(1, 2);

// Check adjacency (O(1))
console.log(adjMatrix[0][1]); // 1
Adjacency List in JavaScript
const numVertices = 3;
const adjList = Array(numVertices).fill().map(() => []);

function addEdgeList(u, v) {
    adjList[u].push(v);
    adjList[v].push(u); // For undirected graph
}

addEdgeList(0, 1);
addEdgeList(1, 2);

// Check adjacency (O(degree(u)))
console.log(adjList[0].includes(1)); // true
Weighted Graph in JavaScript
const weightedAdjList = Array(numVertices).fill().map(() => []);

function addWeightedEdge(u, v, weight) {
    weightedAdjList[u].push({ node: v, weight });
    weightedAdjList[v].push({ node: u, weight }); // For undirected weighted graph
}

addWeightedEdge(0, 1, 5);
addWeightedEdge(1, 2, 10);

9.3. Fundamentals of Graph Traversal

Once a graph structure is established, the next logical task is to systematically visit all its vertices. This process is called graph traversal. There are two primary techniques: Depth-First Search (DFS) and Breadth-First Search (BFS).

The fundamental difference between DFS and BFS lies in the order in which they explore the graph, which stems from the underlying data structure they use to manage yet-to-be-visited vertices:

  1. Depth-First Search (DFS) uses a stack (often implemented via recursion or explicit arrays) to explore as deep as possible along each branch before backtracking. While DFS does not inherently find shortest paths, it is indispensable for tasks like detecting cycles, topological sorting, and exploring a graph when its overall structure is not known in advance.

  2. Breadth-First Search (BFS) uses a queue to explore the graph level by level. It visits all direct neighbors of the starting vertex, then all of their neighbors, and so on. BFS is guaranteed to find the shortest path (in terms of the number of edges) in an unweighted graph.

9.4. Calculation of Paths and Travel Costs from a Given Vertex to All Other Vertices

This subsection examines the problem of finding all possible simple routes between two vertices in a weighted graph. For each route, the algorithm calculates two cumulative characteristics:

  • total distance, measured in kilometers;

  • total travel price, measured in arbitrary units.

The considered task differs from ordinary graph traversal. Classical BFS and DFS are usually used to determine the order of visiting vertices or to find one suitable path. In contrast, the algorithm described here enumerates all possible simple paths from a source vertex to a target vertex and evaluates each of them according to its accumulated weights.

This makes it possible to solve the following practical tasks:

  • find all possible routes between two selected vertices;

  • calculate the total distance of each route;

  • calculate the total travel price of each route;

  • compare routes according to different optimization criteria;

  • select the shortest, cheapest, or otherwise preferable route.

General Idea of the Algorithm

The algorithm is based on recursive backtracking. During the search, the program gradually constructs the current path from the source vertex to the target vertex. At each step, it selects one of the neighboring vertices, adds the corresponding edge characteristics to the accumulated sums, and continues the search recursively.

If the target vertex is reached, the current route is saved as a completed path. After that, the algorithm returns to the previous vertex, restores the former state, and tries another possible continuation.

Thus, the algorithm consists of two alternating phases:

  • forward movement to a neighboring vertex;

  • backward restoration of the previous state.

This mechanism makes it possible to enumerate all simple paths without revisiting vertices already included in the current route.

Functional Structure of the Path Search Module

The path search module is organized as a small group of cooperating functions. Each function performs a clearly separated role in the general recursive process.

The main control function is initPathSearch(). It prepares the initial search context and starts recursion.

The recursive enumeration itself is performed by findPathWithContext(). This function checks the current vertex, processes its neighbors, and controls the transition from one vertex to another.

When the target vertex is reached, the function handleFoundPath() creates a structured description of the completed route and stores it in the resulting collection.

Two auxiliary functions support the backtracking mechanism:

  • prepareMoveToNeighbor() prepares the transition to a neighboring vertex and saves the current state;

  • backtrackFromNeighbor() restores the previous state after the recursive call has finished.

This functional decomposition separates the logic of initialization, recursive search, result processing, forward movement, and backward restoration.

Algorithmic Complexity Analysis

The implemented algorithm enumerates all simple paths between two vertices. Therefore, its running time may grow exponentially with the number of vertices.

In the worst case, especially for a dense graph, the number of possible simple paths can be very large. For this reason, the time complexity can be estimated as factorial or exponential, depending on the graph structure. A conservative upper estimate is:

\$O(V!)\$

where V is the number of vertices.

The space complexity is mainly determined by the recursion depth, the current path, and the visited array. Since a simple path cannot contain more than V vertices, the auxiliary space complexity is:

\$O(V)\$

This approach is suitable for small and medium-sized educational examples. However, for large dense graphs, exhaustive path enumeration becomes impractical. In such cases, specialized algorithms should be used depending on the optimization criterion, for example shortest path algorithms.

Summary

The developed module demonstrates how recursive backtracking can be applied to weighted graphs. Unlike simple traversal algorithms, this approach does not stop after finding one path. Instead, it systematically enumerates all possible simple paths from the source vertex to the target vertex and calculates the quantitative characteristics of each route.

This makes the module useful as an educational example for understanding:

  • recursive graph traversal;

  • path construction;

  • state saving and restoration;

  • backtracking;

  • analysis of weighted routes;

  • comparison of alternative paths.

The same idea can be used as a basis for studying transportation logistics, route planning, and optimization problems.

9.5. Dijkstra’s Algorithm

Dijkstra’s algorithm is designed to find the shortest paths from a given start vertex to all other vertices in a weighted graph, provided that all edge weights are non-negative. This condition is essential: Dijkstra’s algorithm works correctly only when every edge weight is greater than or equal to zero.

The algorithm is named after the renowned Dutch computer scientist Edsger W. Dijkstra, who first conceived it in 1956. Today, Dijkstra’s algorithm is widely used in science, engineering, and everyday applications:

  • GPS navigation systems: finding the fastest routes while taking into account distances and traffic conditions.

  • Telecommunications networks: determining optimal paths for data routing, for example in OSPF and IS-IS protocols.

  • Robotics and games: real-time path planning and object navigation through maps.

Graph Representation for Dijkstra’s Algorithm

For our demonstration of Dijkstra’s algorithm, we will use the graph shown in Figure 9.16. This graph consists of six vertices, labeled from 0 to 5, connected by weighted edges.

Figure 9.16. Target topology setup for Dijkstra shortest path calculation

In JavaScript, this weighted graph can be represented as an object whose keys are vertices and whose values describe adjacent vertices and edge weights:

    0: { 1: 48.3, 2: 40.7, 5: 25.4 },
    1: { 0: 48.3, 2: 37.5, 3: 29.8 },
    2: { 0: 40.7, 1: 37.5, 3: 34.9, 5: 29.8 },
    3: { 1: 29.8, 2: 34.9, 4: 25.4 },
    4: { 3: 25.4, 5: 34.9 },
    5: { 0: 25.4, 2: 29.8, 4: 34.9 }
};

This representation corresponds to an undirected weighted graph. For example, the edge between vertices 0 and 1 has weight 48.3, and therefore it appears in both directions:

0: { 1: 48.3, ... }
1: { 0: 48.3, ... }

How the Algorithm Works

Dijkstra’s algorithm is based on a simple but powerful principle: it iteratively builds the set of vertices for which the shortest distance from the start vertex is already known, and uses these vertices to improve the distance estimates for the remaining vertices.

The mind map of Dijkstra’s algorithm is presented in Figure 9.17.

Figure 9.17. Mind map for Dijkstra’s algorithm

Let us examine the algorithm step by step. We start from vertex 0 and find the shortest paths from this vertex to all other vertices of the graph.

To track its progress, the algorithm maintains three essential data structures:

  1. Distance array (distances or d): for each vertex v, distances[v] stores the current length of the shortest known path from the start vertex s to v. Initially, distances[s] = 0, and distances[v] = Infinity for all other vertices.

  2. Visited array (visited): a marker structure where visited[v] = true means that the shortest distance to vertex v has already been finalized. If visited[v] = false, the vertex is still unprocessed.

  3. Previous array (previous): for each vertex v, previous[v] stores the predecessor of v in the currently known shortest path. This structure is used later to reconstruct the actual shortest paths.

The algorithm can be understood through a city-exploration analogy. Imagine a network of cities connected by roads of different lengths. You start at your home city and repeatedly choose the closest city whose shortest route has not yet been finalized. From that city, you check whether its roads provide shorter routes to neighboring cities.

In the classic implementation shown here, the algorithm uses a linear search to find the unvisited vertex with the smallest current distance. This makes the implementation simple and clear for educational purposes.

Figure 9.21. Marking the selected vertex as processed

The most important operation in the algorithm is edge relaxation. During this step, the algorithm checks whether a shorter path to each neighboring vertex can be obtained through the current vertex.

Figure 9.22. Updating distance estimates for neighboring vertices

For an edge from vertex u to vertex v with weight w(u, v), the relaxation condition is:

distances[u] + w(u, v) < distances[v]

If this condition is true, the algorithm updates the current distance estimate:

distances[v] = distances[u] + w(u, v)

and records the predecessor of vertex v:

previous[v] = u

Thus, relaxation performs two tasks at once:

  • improves the shortest known distance to a vertex;

  • stores information needed to reconstruct the shortest path.

General Control Flow

The overall logic of the algorithm can be described as follows:

  1. Build the list of graph vertices.

  2. Initialize the distances, visited, and previous structures.

  3. Set the distance to the start vertex to 0.

  4. Find the unvisited vertex with the smallest current distance.

  5. Mark this vertex as visited.

  6. Relax all outgoing edges from this vertex.

  7. Repeat the process until there is no reachable unvisited vertex left.

  8. Return the final distances and previous structures.

A JavaScript implementation of the main function can be written as follows:

function dijkstra(graph, startVertex) {
    const vertices = getVerticesList(graph);

    const distances = initializeDistances(vertices, startVertex);
    const visited = initializeVisited(vertices);
    const previous = initializePrevious(vertices);

    let currentVertex = findMinDistanceVertex(distances, visited);

    while (currentVertex !== null) {
        markVertexVisited(visited, currentVertex);
        relaxEdges(graph, currentVertex, distances, previous, visited);

        currentVertex = findMinDistanceVertex(distances, visited);
    }

    return {
        distances: distances,
        previous: previous
    };
}

The returned object contains two results:

  • distances — the lengths of the shortest paths from the start vertex to all reachable vertices;

  • previous — predecessor information that allows the actual shortest paths to be reconstructed.

Complexity

In this classic implementation, the function findMinDistanceVertex performs a linear scan over all vertices. Since it can be called up to V times, where V is the number of vertices, the vertex-selection part has complexity:

O(V^2)

The relaxation of edges contributes:

O(E)

where E is the number of edges.

Therefore, the total time complexity of this classic implementation is:

O(V^2 + E)

For dense graphs, this is often written simply as:

O(V^2)

More advanced implementations use a priority queue or a min-heap to select the next vertex more efficiently, but the classic version shown here is easier to understand and is well suited for explaining the main idea of Dijkstra’s algorithm.
