Section 6. Basic Search Algorithms

Searching is one of the most important algorithms for processing data structures. Search algorithms are designed to check the presence of an element or extract an element from any data structure in which it is stored. More strictly, the search problem can be formulated as follows: find one or more elements in a set, where the elements sought must have a certain property. This property can be absolute or relative. A relative property characterizes an element with respect to other elements: for example, a minimum element in a set of numbers.

There are many search algorithms. Their complexity varies from simple sequential search algorithms to extremely efficient but limited binary search algorithms. Of particular note are algorithms based on presenting the core set of data in a different, more searchable form, which are used in real-world applications for processing data sets in huge databases.

There is no single algorithm to solve all search problems. Some algorithms run faster than others but require additional RAM to run. Others run very quickly but can only be used for pre-sorted arrays. At the same time, the analysis of search algorithms is somewhat different from sorting algorithms. In particular, there is no stability problem for them. However, there may be situations that require the introduction of new criteria of complexity and its assessment. In general, all algorithms are reduced to the following steps:

  • Establishing the property of the elements of the source set; in most cases, these are the values of the elements.

  • Matching the value of the element with the reference property (for absolute properties) or comparing the properties of two elements (for relative properties).

  • Traversing the elements of the set.

In principle, search algorithms differ between search methods and search strategies. Based on the type of search operation, these algorithms are usually classified into two categories:

  • Sequential Search: The list or array is traversed sequentially and each element is checked.

Interval Search: These algorithms are specifically designed for searching in sorted data structures. These types of search algorithms are much more efficient than linear search because they repeatedly target the center of the search structure and divide the search space in half.

This section examines the primary algorithms used for searching data structures. To enhance the understanding of the practical applications of these algorithms, DRAKON-diagrams are provided along with assessments of their time and space complexity.

6.1. Linear Data Searching

a) Linear Search of Unsorted Data

In the case of a linear search for an element in an unordered dataset, for example, in an array or in a slice, the simplest algorithm is to examine all elements until the desired value is found (.Figure 6.1).

Figure 6.1. DRAKON-diagram of the Linear Search Algorithm

This algorithm is not very efficient, but it works on arbitrary collections. Time complexity: O(n). In the worst case (the desired element is in the last position), to find the element you need to traverse all elements. Here "n" is the size of the slice. Space complexity: O(1). No additional memory is required to accommodate the slice. Another worst-case scenario is the absence of the required element.

b) Linear Search of Sorted Data

If the elements of the dataset are sorted in ascending or descending order, finding the desired element is much more efficient than in an unordered linear search. In many cases, you do not have to traverse the whole list. For example, when traversing an ascending sorted list, if an element with a higher value than the search key is encountered, the search is stopped. This approach saves time and increases productivity. .Figure 6.2 shows the DRAKON-diagram of this algorithm.

6.2. Binary Search for Data in a Sorted Structure

Figure 6.2. DRAKON-diagram of the Search in a Sorted Array

Binary search is performed as follows:

  • Determine the value of the element in the middle of the data structure. The resulting value is compared to the value being searched for.

  • If the search value is less than the middle value, the search is carried out in the first half of the elements; otherwise, in the second half.

  • The search continues by determining the value of the middle element in the selected half and comparing it to the key.

  • The process continues until an element with the search value is found or the search interval is empty.

The DRAKON-diagram of the binary search algorithm is represented in Figure 6.3 (the main() module is similar to the previous algorithm):

Figure 6.3. DRAKON-diagram of the Binary Search Algorithm

Time complexity of the binary search algorithm belongs to class O(log n). This means that the asymptotic increase in the time taken by the function to process an input set of size n will not exceed log n. Space complexity: O(1) — no extra space is required.

6.4. Hashing

The time to find an item in a data set often depends on how many comparisons are made. To speed up search and improve computational efficiency, we need fewer comparisons. One way is to map keys to a smaller range of indices – this process is called hashing, and the resulting structure is a hash table.

In terms of abstract data type (ADT) theory, a hash table implements an associative array. It stores key‑value pairs and supports three basic operations: insert (add a pair), lookup (find a value by key), and delete (remove a pair).

From a programming perspective, a hash table consists of buckets – groups of elements whose hash values are equal or close. The hash function transforms a given key into an integer index (the bucket number). Many methods exist to construct a hash function. The simplest is the division method, where the hash is the remainder of the key divided by the number of buckets:
h(key) = key mod m (assuming the key is an integer). In JavaScript, for numeric keys, this becomes h = x % m.

6.4.1 Creating a Hash Table and Chain Nodes

To implement a hash table using chaining, we need two helper structures: the table itself (an array of buckets) and a node for the linked list.

Creating the Hash Table

The function createHashTable(size) initializes a new hash table instance. It takes one argument – the desired number of buckets. Internally, it does the following:

  • Creates an array table of the given length.

  • Fills each element of the array with null. This means all buckets are initially empty – no key has been added yet.

  • Returns an object containing table and size. This object will be used by all subsequent operations (insert, traverse, lookup, remove).

a) Function for Creating a Hash Table (createHashTable)

In practice, the table object often also includes a count field (current number of elements) and a loadFactor (threshold for resizing). However, we start with a minimal version for simplicity.

Creating a Node for the Chain

Because collisions are resolved with linked lists, each element in a bucket must store not only a key‑value pair but also a reference to the next element in the same chain.

The function createNode(key, value, next = null) creates such a node. It:

  • Accepts a key, a value, and an optional reference to the next node (default null).

  • For debugging, prints a message to the console indicating that a node has been created. In production code this log is usually removed, but in educational examples it helps trace the algorithm’s behavior.

  • Returns an object with three fields: key, value, and next.

b) Function for Creating a node (createNode)

Note that the next reference allows us to build singly linked lists inside a bucket. If only one element exists in a bucket, its next is null. When we add a new element that hashes to the same index, we place it at the beginning (or end – depending on the strategy; our algorithm will add to the head for simplicity).

As a result, this hash table would have as many singly linked lists (buckets) as specified by the Size constant. In the example below, the number of slots is 15.

The algorithm implementing the remainder method sequentially processes each element from the original set, dividing it by 15 and returning the remainder as the hash value. This can be expressed as h(item) = item mod 15. Thus, the remainder method converts a collection of 120 integers into a hash table of 15 slots. The search for an element is now greatly accelerated as it takes two steps: first, the hash function h = (x % m) computes the hash index, and then the search is performed within a bucket (approximately 7–8 elements).
The implementation of the basic hash table functions using the hash function is as follows:

  • Create a HashTable structure of size m to store objects.

  • Compute the object’s hash code by passing it through the hash function.

  • Obtain the bucket hash indices where the objects will be stored.

  • Store these objects in the designated bucket.

Another example of a "good" hash function for use with integer key values is the mean square method. The mean square method squares the key value and then extracts the middle digits of the result, giving a value in the range from 0 to M. A software implementation of a hash function algorithm in Go would involve using built-in functions for converting integers to strings (strconv.Atoi(i)) and vice versa (strconv.Itoa(i)). For example, for any four-digit number, the hash function is:

func hFunc(i int) int {
var j int
var s string
i = i * i
s = strconv.Itoa(i)
s = s[3:5]
j, _ = strconv.Atoi(s)
return j
}

Topics Beyond the Scope of This Book

This section has focused on the fundamental principles of hashing and the implementation of a hash table using separate chaining. However, several important and advanced aspects of modern hashing remain outside the scope of this book:

Open addressing techniques — linear probing, quadratic probing, and double hashing.
Dynamic resizing of hash tables when the load factor exceeds a threshold.
Cryptographic hash functions (MD5, SHA family, etc.) used for data integrity and security rather than indexing.
Perfect hashing and minimal perfect hashing, which eliminate collisions entirely but require complex precomputation.
Distributed hash tables (DHTs) — used in peer-to-peer networks and blockchain architectures.

These topics will be explored in future works devoted to large-scale data management, high-performance systems, and secure information storage.
